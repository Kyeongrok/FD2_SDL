# 용의기사2 DAT 파일 구조 분석 보고서

## 개요

본 문서는 IDA Pro 역공학을 통한 FD2.EXE 심층 분석을 기반으로, 각 DAT 파일의 데이터 구조를 검증하고 보완한다. 모든 DAT 파일은 통일된 인덱스 테이블 + 데이터 블록 저장 방식을 사용하지만, 구체적인 인덱스 구조는 파일마다 다르다.

---

## 공통 파일 포맷

### 공통 헤더 구조 (모든 DAT 파일)

```
+------------------+ <- 0
| 매직 "LLLLLL"   |  6바이트
+------------------+ <- 6
| 인덱스 항목 수(X) |  4바이트 (little-endian uint32)
+------------------+ <- 10
| 리소스 오프셋 테이블 |  각 항목 8바이트: [시작 오프셋, 끝 오프셋]
| [start, end]     |
| [start, end]     |
| ...              |
+------------------+ <- 10 + X*8
| 리소스 데이터 영역   |
+------------------+
```

### 핵심 로드 함수 분석

DAT 파일 읽기의 핵심 함수인 `sub_111BA` 분석을 통해 파일 구조를 검증했다:

```c
// IDA 디컴파일 코드 (주소 0x111BA)
void __fastcall sub_111BA(..., int a7, ...) {
    fseek(_rb_, 4 * a7 + 6, 0);   // 인덱스 위치로 탐색: 4*인덱스번호 + 6
    sub_373CA(v8, 1u, 8, _rb_);   // 8바이트 읽기
    v9 = *v8;                      // 시작 오프셋 (startOffset)
    dword_53BFF = v8[1] - *v8;   // 크기 = 끝 오프셋 - 시작 오프셋
    
    // 메모리 할당 및 데이터 읽기
    fseek(_rb_, v9, 0);            // 데이터 시작 위치로 점프
    sub_373CA(v10, 1u, dword_53BFF, _rb_); // 데이터 블록 읽기
}
```

**핵심 검증 포인트**:
- 인덱스 위치 계산식: `offset = 4 * resourceIndex + 6`
- 각 인덱스 항목: 8바이트 = 4바이트 시작 오프셋 + 4바이트 끝 오프셋
- 데이터 블록 크기: `size = endOffset - startOffset`

## RLE 압축 포맷 (검증 완료)

역공학 코드로 검증된 RLE 포맷:

| 제어 코드 상위 2비트 | 모드 | 의미 |
|------------|------|------|
| 00 | Fill | 다음 바이트로 (num & 0x3F) + 1 번 채우기 |
| 01 | Literal | (num & 0x3F) + 1 바이트 직접 복사 |
| 10 | Fill | 00 과 동일 |
| 11 | Skip | 포인터만 (num & 0x3F) + 1 바이트 이동 |

**압축 해제 알고리즘** (C 코드):

```c
void decompress_rle(byte* src, int src_size, byte* dst, int stride) {
    word count = *(word*)src;      // 행당 바이트 수
    word num_rows = *(word*)(src + 2);  // 행 개수
    
    src += 4;
    int pos = 0;
    
    for (int row = 0; row < num_rows; row++) {
        int remaining = count;
        
        while (remaining > 0 && pos < src_size) {
            byte code = src[pos++];
            int num = (code & 0x3F) + 1;
            if (num > remaining) num = remaining;
            
            int type = (code >> 6) & 0x3;
            
            switch(type) {
                case 0: // 00: fill
                case 2: // 10: fill
                    byte fill = src[pos++];
                    memset(dst, fill, num);
                    dst += num;
                    break;
                case 1: // 01: literal
                    memcpy(dst, src + pos, num);
                    pos += num;
                    dst += num;
                    break;
                case 3: // 11: skip
                    dst += num;
                    break;
            }
            remaining -= num;
        }
        dst += stride - count;
    }
}
```

## 검증 방법

IDA Pro MCP 로 FD2.EXE 의 핵심 함수들을 분석해 검증:

1. **sub_111BA** (0x111BA): 범용 DAT 파일 로드 함수
   - 인덱스 위치 공식 확인: `4 * index + 6`
   - 항목당 8바이트 포맷 확인: `[start, end]`

2. **문자열 참조** (0x51A40-0x51A79):
   - DAT 파일명 리스트 확인

3. **main 함수** (0x25BF4): 각 로드 함수 호출

## 결론

IDA 역공학 검증을 통해 다음을 확인했다:
1. 모든 DAT 파일은 통일된 "LLLLLL" 매직과 인덱스 테이블 구조 사용
2. 인덱스 위치 공식: `offset = 4 * resourceIndex + 6`
3. 각 인덱스 항목 8바이트: [시작 오프셋, 끝 오프셋]
4. 오프셋 6 의 값(X)은 인덱스 테이블 항목 수를 나타냄
5. 실제 리소스 수는 인덱스 항목 수보다 적을 수 있음

대부분의 파일은 이미지 리소스를 성공적으로 파싱할 수 있으며, 나머지 파일은 고유의 서브 인덱스 구조에 대한 추가 분석이 필요하다.
