# FD2.EXE 启动过程动态调试必要性分析

> 分析静态逆向的局限性，确定必须通过动态调试才能获取的信息

---

## 结论摘要

**是的，存在必须通过动态调试才能完整逆向的部分。** 静态分析已覆盖约 70% 的启动流程，但以下 30% 的区域需要动态调试补充：

| 类别 | 严重程度 | 影响范围 |
|------|----------|----------|
| 反编译失败的函数 | 🔴 高 | 6个关键初始化函数 |
| JUMPOUT 跳转目标 | 🟡 中 | 400+ 处未解析的跳转 |
| 全局变量语义 | 🟡 中 | 200+ 个全局变量含义不明 |
| 寄存器调用约定 | 🟠 低-中 | Watcom register calling |
| 硬件中断交互 | 🟡 中 | int386 调用的具体行为 |

## 一、必须动态调试的部分

### 1.1 反编译失败的启动函数 (🔴 高优先级)

以下函数在启动过程中被调用，但 IDA 反编译失败：

| 地址 | 函数 | 调用者 | 原因 | 必须调试的原因 |
|------|------|--------|------|----------------|
| 0x244B6 | sub_244B6 | main | 返回None | 可能是视频模式初始化 |
| 0x2548C | sub_2548C | main | 返回None | 可能是内存管理器设置 |
| 0x25757 | sub_25757 | main | 返回None | 可能是中断向量设置 |
| 0x336A0 | sub_336A0 | main, sub_25EBB | 返回None | 状态处理函数 |
| 0x33AF1 | sub_33AF1 | main, sub_25EBB | 返回None | 状态处理函数 |
| 0x33C9D | sub_33C9D | main, sub_25EBB | 返回None | 状态处理函数 |
| 0x16F55 | sub_16F55 | sub_117E7 | 返回None | 菜单选择处理 |

**调试建议**：
```bash
# 在这些函数入口设置断点
b 0x244B6
b 0x2548C  
b 0x25757
b 0x336A0
b 0x33AF1
b 0x33C9D
b 0x16F55

# 运行并观察:
# 1. 函数被调用时的参数 (寄存器 eax, edx, ecx, ebx)
# 2. 函数返回值
# 3. 函数执行的副作用 (内存写入、端口操作)
```

### 1.2 JUMPOUT 跳转目标 (🟡 中优先级)

反编译代码中存在 **400+ 个 JUMPOUT** 语句，无法确定跳转目标：

**影响的启动函数**：
- `main` - JUMPOUT(0x22BBE)
- `sub_1F894` - JUMPOUT(0x13994)
- `sub_10010` - JUMPOUT(0x22BBE)
- `sub_25EBB` - 多处 JUMPOUT
- 所有音频相关函数

**典型问题代码**：
```c
// sub_1F894 末尾
JUMPOUT(0x13994);  // 跳转到哪里？不知道！

// sub_25977 音乐控制
JUMPOUT(0x39162);  // 跳转到清理代码？还是错误处理？
JUMPOUT(0x3916A);  // 未知目标
```

**调试建议**：
```bash
# 在 JUMPOUT 语句前设置断点
b *0x13993  # sub_1F894 JUMPOUT前
b *0x22BBD  # main JUMPOUT前

# 运行并单步跟踪
# 观察 EIP 寄存器确定实际跳转目标
```

### 1.3 全局变量语义不明 (🟡 中优先级)

**200+ 个全局变量**只有地址，没有语义标注：

| 变量 | 地址 | 已知用途 | 不确定部分 |
|------|------|----------|------------|
| n17 | - | 当前游戏状态索引 | 有效值范围、状态机映射 |
| n6_0 | - | 单位数量 | 最大值、数据结构布局 |
| dword_53A45 | 0x53A45 | 单位数据数组 | 每个单位80字节的具体字段 |
| dword_53A55 | 0x53A55 | 存档主数据 | 2211字节的具体布局 |
| byte_51AAC | - | 菜单锁定标志 | 何时设置、影响哪些操作 |
| dword_53BFB | - | 游戏状态 | 与n17的关系 |

**调试建议**：
```bash
# 监控全局变量访问
# 在变量地址设置内存断点
b *0x53A45  # 监控单位数组访问

# 或使用数据断点 (hardware watchpoint)
watch dword [0x53BFB]
```

### 1.4 Watcom 寄存器调用约定 (🟠 低-中优先级)

IDA 对 Watcom register calling convention 的解析不完整：

```c
// 原始反编译 (不确定参数位置)
int __usercall sub_25977@<eax>(
    __int32 a1@<eax>,    // 是参数吗？还是返回值？
    int a2@<edx>,        // 第二个参数？
    int a3@<ecx>,        // 第三个参数？
    int n99@<ebx>,       // 这个呢？
    unsigned __int8 *a6@<edi>  // 指针参数？
);

// 实际参数可能不同
```

**调试建议**：
```bash
# 在函数入口设置断点，观察寄存器值
b *0x25977
run
info registers eax edx ecx ebx esi edi

# 在函数调用前后检查参数
# 验证哪些是输入参数，哪些是临时值
```

### 1.5 DOS 中断调用 (🟡 中优先级)

`int386` 调用的具体行为需要动态验证：

```c
// sub_11019 中的中断调用
int386(22, &regs, &regs);  // 22h = 键盘中断？
// regs 结构体的内容和返回值是什么？

// sub_10010 中
int386(16, &regs, &regs);  // 16h = 系统服务？
// 获取什么系统信息？
```

**调试建议**：
```bash
# 在 int386 调用前后设置断点
b *0x11019  # 或具体中断调用地址
# 单步执行，观察：
# 1. 调用前的寄存器状态
# 2. 中断号 (AH/AL)
# 3. 调用后的返回值
```

## 三、动态调试方案

### 3.1 调试工具选择

| 工具 | 适用场景 | 优势 |
|------|----------|------|
| DOSBox Debugger | DOS程序调试 | 原生DOS环境，支持中断调试 |
| Turbo Debugger (TD) | 传统DOS调试 | Watcom兼容性好 |
| IDA Pro Remote Debug | IDA集成调试 | 与静态分析结合 |
| x64dbg + DOS插件 | 现代调试器 | 现代界面，脚本支持 |

### 3.2 推荐调试流程

```
1. 启动阶段 (0x0000 - main)
   ├─ 断点: start, __CMain, main
   ├─ 观察: 栈初始化、全局变量清零
   
2. 音频初始化 (main前半部分)
   ├─ 断点: sub_37D3E, sub_3AA72, sub_3908B
   ├─ 观察: 驱动句柄返回值、错误码
   
3. DAT文件加载 (main中间部分)
   ├─ 断点: sub_111BA (多次调用)
   ├─ 观察: 文件句柄、缓冲区地址、资源大小
   
4. 存档检查/开场动画 (sub_25EBB → sub_1F894)
   ├─ 断点: sub_25EBB入口, sub_1F894入口
   ├─ 观察: 存档文件读取、动画帧循环
   
5. 菜单等待输入 (sub_1F894后半部分)
   ├─ 断点: 键盘中断调用
   ├─ 观察: 按键扫描码、菜单选项处理
```

### 3.3 关键调试断点清单

```bash
# 必须断点 (反编译失败)
b *0x244B6
b *0x2548C
b *0x25757
b *0x336A0
b *0x33AF1
b *0x33C9D

# 重要断点 (验证静态分析)
b *0x25BF4        # main 入口
b *0x1F894        # 开场动画
b *0x10010        # 存档加载
b *0x111BA        # DAT加载器 (可设置条件断点)
b *0x25EBB        # 状态检查

# JUMPOUT 目标断点
b *0x22BBE        # main JUMPOUT目标
b *0x13994        # sub_1F894 JUMPOUT目标
b *0x39162        # 音频函数 JUMPOUT目标
b *0x3916A        # 音频函数 JUMPOUT目标
```

## 五、总结

| 问题类型 | 静态分析覆盖 | 需要动态调试 |
|----------|--------------|--------------|
| 函数功能 | ~70% | 30% (6个失败函数) |
| 控制流 | ~60% | 40% (JUMPOUT未解析) |
| 数据结构 | ~50% | 50% (布局不明) |
| 全局变量 | ~40% | 60% (语义不明) |
| **总体** | **~60%** | **~40%** |

**结论**：动态调试对于完整理解 FD2 启动过程是**必须的**，主要用于：
1. 补全 6 个反编译失败的函数
2. 解析 400+ 个 JUMPOUT 跳转目标  
3. 验证数据结构布局和全局变量含义
4. 理解 Watcom 寄存器调用约定的参数传递

预计需要 **4-8 小时**的动态调试工作才能达到 90%+ 的理解度。
