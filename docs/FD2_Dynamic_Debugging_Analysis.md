# FD2.EXE 시작 과정 동적 디버깅 필요성 분석

> 정적 역공학의 한계를 분석하고, 동적 디버깅을 통해서만 얻을 수 있는 정보를 확정

---

## 결론 요약

**네, 동적 디버깅을 통해서만 완전한 역공학이 가능한 부분이 존재합니다.** 정적 분석은 시작 흐름의 약 70%를 커버했지만, 다음 30% 영역은 동적 디버깅으로 보완해야 합니다:

| 분류 | 심각도 | 영향 범위 |
|------|----------|----------|
| 디컴파일 실패 함수 | 🔴 높음 | 6개의 핵심 초기화 함수 |
| JUMPOUT 점프 대상 | 🟡 중간 | 400개 이상의 미해결 점프 |
| 전역 변수 의미 | 🟡 중간 | 200개 이상의 전역 변수 의미 불명 |
| 레지스터 호출 규약 | 🟠 낮음-중간 | Watcom register calling |
| 하드웨어 인터럽트 상호작용 | 🟡 중간 | int386 호출의 구체적 동작 |

## 1. 동적 디버깅이 반드시 필요한 부분

### 1.1 디컴파일 실패 시작 함수 (🔴 높은 우선순위)

다음 함수들은 시작 과정에서 호출되지만 IDA 디컴파일에 실패했습니다:

| 주소 | 함수 | 호출자 | 원인 | 디버깅 필요 이유 |
|------|------|--------|------|----------------|
| 0x244B6 | sub_244B6 | main | None 반환 | 비디오 모드 초기화일 가능성 |
| 0x2548C | sub_2548C | main | None 반환 | 메모리 매니저 설정일 가능성 |
| 0x25757 | sub_25757 | main | None 반환 | 인터럽트 벡터 설정일 가능성 |
| 0x336A0 | sub_336A0 | main, sub_25EBB | None 반환 | 상태 처리 함수 |
| 0x33AF1 | sub_33AF1 | main, sub_25EBB | None 반환 | 상태 처리 함수 |
| 0x33C9D | sub_33C9D | main, sub_25EBB | None 반환 | 상태 처리 함수 |
| 0x16F55 | sub_16F55 | sub_117E7 | None 반환 | 메뉴 선택 처리 |

**디버깅 제안**:
```bash
# 이 함수들의 진입점에 중단점 설정
b 0x244B6
b 0x2548C  
b 0x25757
b 0x336A0
b 0x33AF1
b 0x33C9D
b 0x16F55

# 실행 후 관찰:
# 1. 함수 호출 시점의 인자 (레지스터 eax, edx, ecx, ebx)
# 2. 함수 반환값
# 3. 함수 실행의 부수 효과 (메모리 쓰기, 포트 연산)
```

### 1.2 JUMPOUT 점프 대상 (🟡 중간 우선순위)

디컴파일된 코드에 **400개 이상의 JUMPOUT** 문이 존재하지만 점프 대상을 확정할 수 없습니다:

**영향 받는 시작 함수**:
- `main` - JUMPOUT(0x22BBE)
- `sub_1F894` - JUMPOUT(0x13994)
- `sub_10010` - JUMPOUT(0x22BBE)
- `sub_25EBB` - 여러 곳에 JUMPOUT
- 모든 오디오 관련 함수

**전형적인 문제 코드**:
```c
// sub_1F894 끝 부분
JUMPOUT(0x13994);  // 어디로 점프? 알 수 없음!

// sub_25977 음악 제어
JUMPOUT(0x39162);  // 정리 코드로 점프? 아니면 에러 처리?
JUMPOUT(0x3916A);  // 대상 불명
```

**디버깅 제안**:
```bash
# JUMPOUT 문 직전에 중단점 설정
b *0x13993  # sub_1F894 JUMPOUT 직전
b *0x22BBD  # main JUMPOUT 직전

# 실행 후 단일 스텝 추적
# EIP 레지스터를 관찰해 실제 점프 대상 확인
```

### 1.3 전역 변수 의미 불명 (🟡 중간 우선순위)

**200개 이상의 전역 변수**가 주소만 있고 의미 주석이 없습니다:

| 변수 | 주소 | 알려진 용도 | 불확실한 부분 |
|------|------|----------|------------|
| n17 | - | 현재 게임 상태 인덱스 | 유효 값 범위, 상태 머신 매핑 |
| n6_0 | - | 유닛 개수 | 최대치, 데이터 구조 레이아웃 |
| dword_53A45 | 0x53A45 | 유닛 데이터 배열 | 유닛당 80바이트의 구체 필드 |
| dword_53A55 | 0x53A55 | 세이브 메인 데이터 | 2211바이트의 구체 레이아웃 |
| byte_51AAC | - | 메뉴 잠금 플래그 | 언제 설정, 어떤 동작에 영향 |
| dword_53BFB | - | 게임 상태 | n17 와의 관계 |

**디버깅 제안**:
```bash
# 전역 변수 접근 모니터링
# 변수 주소에 메모리 중단점 설정
b *0x53A45  # 유닛 배열 접근 감시

# 또는 데이터 중단점 사용 (hardware watchpoint)
watch dword [0x53BFB]
```

### 1.4 Watcom 레지스터 호출 규약 (🟠 낮음-중간 우선순위)

IDA 의 Watcom register calling convention 파싱이 불완전합니다:

```c
// 원본 디컴파일 (인자 위치 불확정)
int __usercall sub_25977@<eax>(
    __int32 a1@<eax>,    // 인자인가? 아니면 반환값?
    int a2@<edx>,        // 두 번째 인자?
    int a3@<ecx>,        // 세 번째 인자?
    int n99@<ebx>,       // 이건 뭐지?
    unsigned __int8 *a6@<edi>  // 포인터 인자?
);

// 실제 인자는 다를 수 있음
```

**디버깅 제안**:
```bash
# 함수 진입점에 중단점 설정, 레지스터 값 관찰
b *0x25977
run
info registers eax edx ecx ebx esi edi

# 함수 호출 전후로 인자 검증
# 입력 인자와 임시 값을 구별
```

### 1.5 DOS 인터럽트 호출 (🟡 중간 우선순위)

`int386` 호출의 구체적 동작은 동적 검증이 필요합니다:

```c
// sub_11019 안의 인터럽트 호출
int386(22, &regs, &regs);  // 22h = 키보드 인터럽트?
// regs 구조체의 내용과 반환값은?

// sub_10010 안
int386(16, &regs, &regs);  // 16h = 시스템 서비스?
// 어떤 시스템 정보를 획득?
```

**디버깅 제안**:
```bash
# int386 호출 전후에 중단점 설정
b *0x11019  # 또는 구체적 인터럽트 호출 주소
# 단일 스텝 실행, 관찰:
# 1. 호출 전 레지스터 상태
# 2. 인터럽트 번호 (AH/AL)
# 3. 호출 후 반환값
```

다음 부분들은 IDA 디컴파일로 충분히 이해되었으며, 동적 디버깅이 필요하지 않습니다:

## 3. 동적 디버깅 방안

### 3.1 디버깅 도구 선택

| 도구 | 적용 시나리오 | 장점 |
|------|----------|------|
| DOSBox Debugger | DOS 프로그램 디버깅 | 네이티브 DOS 환경, 인터럽트 디버깅 지원 |
| Turbo Debugger (TD) | 전통 DOS 디버깅 | Watcom 호환성 우수 |
| IDA Pro Remote Debug | IDA 통합 디버깅 | 정적 분석과 결합 |
| x64dbg + DOS 플러그인 | 현대 디버거 | 현대적 UI, 스크립트 지원 |

### 3.2 추천 디버깅 흐름

```
1. 시작 단계 (0x0000 - main)
   ├─ 중단점: start, __CMain, main
   ├─ 관찰: 스택 초기화, 전역 변수 클리어
   
2. 오디오 초기화 (main 전반부)
   ├─ 중단점: sub_37D3E, sub_3AA72, sub_3908B
   ├─ 관찰: 드라이버 핸들 반환값, 에러 코드
   
3. DAT 파일 로드 (main 중반부)
   ├─ 중단점: sub_111BA (여러 번 호출됨)
   ├─ 관찰: 파일 핸들, 버퍼 주소, 리소스 크기
   
4. 세이브 체크 / 오프닝 애니메이션 (sub_25EBB → sub_1F894)
   ├─ 중단점: sub_25EBB 진입, sub_1F894 진입
   ├─ 관찰: 세이브 파일 읽기, 애니메이션 프레임 루프
   
5. 메뉴 입력 대기 (sub_1F894 후반부)
   ├─ 중단점: 키보드 인터럽트 호출
   ├─ 관찰: 키 스캔 코드, 메뉴 옵션 처리
```

### 3.3 핵심 디버깅 중단점 목록

```bash
# 필수 중단점 (디컴파일 실패)
b *0x244B6
b *0x2548C
b *0x25757
b *0x336A0
b *0x33AF1
b *0x33C9D

# 중요 중단점 (정적 분석 검증)
b *0x25BF4        # main 진입
b *0x1F894        # 오프닝 애니메이션
b *0x10010        # 세이브 로드
b *0x111BA        # DAT 로더 (조건부 중단점 가능)
b *0x25EBB        # 상태 체크

# JUMPOUT 대상 중단점
b *0x22BBE        # main JUMPOUT 대상
b *0x13994        # sub_1F894 JUMPOUT 대상
b *0x39162        # 오디오 함수 JUMPOUT 대상
b *0x3916A        # 오디오 함수 JUMPOUT 대상
```

## 4. 검증 체크리스트

동적 디버깅을 통해 검증해야 할 핵심 문제:

### 4.1 기능 검증
- [ ] sub_244B6, sub_2548C, sub_25757 의 구체 기능
- [ ] 모든 JUMPOUT 의 실제 점프 대상
- [ ] int386(16, ...) 이 획득하는 시스템 정보 내용
- [ ] int386(22, ...) 키보드 인터럽트의 스캔 코드 포맷

### 4.2 데이터 구조 검증
- [ ] 세이브 파일 (22987바이트) 의 완전한 구조
- [ ] 유닛 데이터 (유닛당 80바이트) 의 필드 의미
- [ ] FDMUS.DAT 음악 데이터 포맷
- [ ] FDICON.B24 아이콘 데이터 구조

### 4.3 상태 머신 검증
- [ ] n17 (게임 상태) 의 유효 값 범위
- [ ] byte_51E63[] 배열의 상태-음악 매핑
- [ ] 상태 전환 트리거 조건

## 5. 총괄

| 문제 유형 | 정적 분석 커버리지 | 동적 디버깅 필요 |
|----------|--------------|--------------|
| 함수 기능 | ~70% | 30% (6개 실패 함수) |
| 제어 흐름 | ~60% | 40% (JUMPOUT 미해결) |
| 데이터 구조 | ~50% | 50% (레이아웃 불명) |
| 전역 변수 | ~40% | 60% (의미 불명) |
| **전체** | **~60%** | **~40%** |

**결론**: FD2 시작 과정의 완전한 이해를 위해 동적 디버깅은 **필수**이며, 주로 다음 용도로 사용됩니다:
1. 디컴파일 실패한 6개 함수 보완
2. 400개 이상의 JUMPOUT 점프 대상 해석
3. 데이터 구조 레이아웃과 전역 변수 의미 검증
4. Watcom 레지스터 호출 규약의 인자 전달 이해

90% 이상의 이해도에 도달하려면 약 **4-8시간**의 동적 디버깅 작업이 필요합니다.
